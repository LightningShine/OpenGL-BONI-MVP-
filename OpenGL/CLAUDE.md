# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## О проекте

RAJAGP Client — Windows-клиент (x64 / ARM64) хронометража и телеметрии для гонок RC-машин и картинга в реальном времени. C++17, OpenGL 3.3 (glad) + GLFW + Dear ImGui, GeographicLib (UTM-геодезия), WinHTTP WebSocket, serialib (COM-порты). Подробное описание функционала и использования — в корневом `README.md`.

## Сборка

Solution `OpenGL.sln` в корне репозитория, один проект `OpenGL/OpenGL.vcxproj`, тулсет **v145**, C++17.

```powershell
msbuild OpenGL.sln /p:Configuration=Release /p:Platform=ARM64
```

- **На этой машине (Windows on ARM) собирать и запускать только ARM64, не x64** — x64-сборка с arm64-зависимостями vcpkg падает при старте с 0xc000007b.
- Бинарь: `ARM64\Release\OpenGL.exe` (или `x64\Release\OpenGL.exe`).
- Зависимости из vcpkg в `C:\vcpkg` (`glfw3`, `geographiclib`); триплет выбирается автоматически по платформе.
- **Жёсткое требование**: рядом с этим репозиторием должен лежать закрытый репозиторий `RAJAGP Server` (`..\..\RAJAGP Server\` относительно vcxproj). Клиент компилирует из него исходники протокола `rajagp_core` (`core\src\Crc.cpp`, `core\src\RajaParser.cpp`) и подключает `core\include`. Без него сборка не пройдёт.
- Автотестов и линтера в проекте нет.

## Архитектура

Весь код — в `OpenGL/`. Точка входа и главный цикл: `src/core/main.cpp`.

### Источники данных → пайплайн телеметрии

Три независимых способа получения данных о машинах, все сходятся в один пайплайн (`processIncomingTelemetry` → глобальный `g_vehicles`):

1. **COM / serial** (`src/network/ESP32_Code.*`) — приёмник SX1280/LoRa на USB COM-порту; порты обнаруживаются автоматически в фоновом потоке. Формат пакетов общий с сервером через `rajagp_core` (RajaParser).
2. **Track Server** (`src/network/TrackServerClient.*`) — WebSocket-клиент на нативном WinHTTP (намеренно без сторонних сокет-библиотек, чтобы работало на ARM64). Свой поток соединения с реконнектом. Сервер присылает роль (user/admin), геометрию трека, состояния машин и **авторитетные** тайминги (позиция/круг/лучший), которые перекрывают локальный расчёт. Отслеживает качество канала (потери, джиттер).
3. **Симулятор** (`src/network/SimulationServer.*`) — генерирует машины на загруженном треке для тестов без железа.

### Правило потоков

Сетевые/serial-потоки никогда не трогают OpenGL. Данные переходят в рендер-поток через паттерн pending-consume — например, геометрия трека с сервера применяется в рендер-потоке через `TrackServerClient::consumePendingTrack()`, потому что загрузка на GPU должна происходить там. Общие данные — под мьютексами (`g_track_mutex`, `points_mutex` и т.п.).

### Хронометраж

`src/racing/RaceManager.*` считает круги, секторы, дельты и позиции локально по положению машин (для COM/симулятора). При подключении к Track Server тайминги сервера авторитетны и пишутся в `g_vehicles` (`serverPosition` = 0 означает локальную сессию). Режимы гонки (Circuit/Time Attack/Rally) и фазы (Practice → Race → Finishing → Finished) — в `src/racing/ModeManager/` и `StopReset/`.

### UI — важная ловушка

- `OpenGL/UI.cpp`, `UI.h`, `UI_Elements.cpp`, `UI_Elements.h` в **корне проекта** — это реальные компилируемые файлы главного UI (большие, легаси).
- `OpenGL/src/ui/UI.cpp` и `src/ui/UI.h` существуют, но **не включены в vcxproj — их правка ничего не меняет**. При сомнениях проверяйте `<ClCompile>` в `OpenGL.vcxproj`.
- Новый UI — модульный в `src/ui/`: `pro/` (PRO-экран Pit Wall — ProView + по файлу на плавающую панель: Laptime, LapList, LapInfo, Sectors, TrackMap, GForce, Channels, Events), `UIRaceManager/` (табло, флаги, статус-бар), `Accounts` (админ-токены), `ui_scale.*` (DPI-масштабирование по мониторам с пересборкой шрифтов при смене монитора).
- Масштабы каналов панелей хранятся в `pro_scales.ini`, раскладка ImGui — в `imgui.ini` (оба в `OpenGL/`, отслеживаются git и часто «грязные» — обычно не предназначены для коммита).

### Треки

`src/track/` — два пути создания: запись одного круга по телеметрии (`TrackRecorder`, замыкание петли и сглаживание сплайном в `TelemetryTrackBuilder`) и двухконтурный алгоритм dual-edge (проезд по левому краю, затем по правому → трек с реальными границами). Dual-edge треки хранятся в бинарном `.trk2` (заголовок + два полилайна, UTM-привязка через GeographicLib) в `saves/`; трек также передаётся с сервера чанками. Конвенции систем координат (txt/trk2/sim, render offset) тонкие — перед изменением чего-либо позиционного смотрите `src/rendering/Render.*` и `TrackEditor`.

### Сторонние библиотеки

Вендоренные библиотеки — в `libraries/include/` (ImGui, заголовки GLFW и GeographicLib, serialib, stb, nlohmann/json) — их не редактируем. `src/thirdparty/glad.c` — сгенерированный файл.

# Свод правил написания кода RAJAGP

## 0. Как пользоваться этим документом

Правила обязательны для людей и для ИИ. ИИ-агент получает этот файл как контекст перед генерацией кода. Форматирование (отступы, скобки, длина строк) сюда не входит — оно задано файлом `.clang-format` и применяется автоматически, о нём не думают.

Принцип над всеми правилами: **код читают в 10 раз чаще, чем пишут.** Пишем для того, кто откроет файл через год и ничего не помнит. Этот кто-то — вы.

---

## 1. Именование

Английский для всех идентификаторов. Русский — только в комментариях и логах.

- **Переменные, функции** — `snake_case`: `lap_time`, `parse_packet()`, `car_position`.
- **Классы, структуры, enum-типы** — `PascalCase`: `TrackServer`, `DataSource`, `PacketParser`.
- **Константы, значения enum** — `UPPER_CASE`: `MAX_DEVICES`, `SLOT_ID`, `MAGIC_MARKER`.
- **Приватные поля класса** — с суффиксом `_`: `socket_`, `buffer_`, `last_fix_`. Сразу видно «это состояние объекта, а не локальная переменная».
- **Файлы** — `snake_case`: `packet_parser.cpp`, `data_source.hpp`.

Имя говорит **что это**, а не какого оно типа. `car_count`, а не `int_cars`. Не сокращаем до неузнаваемости: `receiver`, а не `rcv`. Исключение — общепринятое в вашей области: `crc`, `gps`, `rtk`, `id`, `utc` понятны всем.

```cpp
// плохо
int n;                    // что за n?
float t;                  // время? температура?
auto d = get();           // что вернули?

// хорошо
int device_count;
float lap_time_seconds;   // единица прямо в имени — спасает от багов
DataSnapshot snapshot = get_latest_snapshot();
```

Единицы измерения в имя, если не очевидны: `timeout_ms`, `distance_meters`, `lap_time_seconds`. Это дёшево и убивает целый класс ошибок.

## 2. Безопасная работа с памятью (сердце свода)

Главное правило современного C++: **вы почти никогда не пишете `new` и `delete` вручную.** Ручное управление памятью — источник большинства падений и утечек. Вместо этого — три инструмента.

**RAII** — ресурс освобождается сам, когда переменная выходит из области видимости. Это фундамент. Открыли файл/сокет/буфер в объекте — он закроется в деструкторе автоматически, даже если случилась ошибка. Вам не нужно помнить «закрыть».

**Умные указатели** вместо голых:

- `std::unique_ptr<T>` — по умолчанию. «Этим объектом владеет ровно один хозяин». Когда хозяин исчезает — объект удаляется сам.
- `std::shared_ptr<T>` — только когда владельцев реально несколько. Дороже, не по умолчанию.
- Голый указатель `T*` — **только как "смотрю, но не владею"**, и никогда не вызываем на нём `delete`.

```cpp
// плохо — ручное управление, забыл delete = утечка, ошибка между = утечка
Receiver* r = new Receiver();
r->start();
delete r;                 // а если start() кинул исключение? delete не вызовется

// хорошо — удалится сам, при любом выходе
auto receiver = std::make_unique<Receiver>();
receiver->start();
// конец функции — receiver освобождён автоматически
```

**Кто чем владеет — писать явно.** Для каждого объекта в дизайне ответьте «кто его хозяин и когда он умрёт». Если не можете ответить — дизайн ещё не готов, код писать рано.

Важная оговорка про GPU (для client): умные указатели **не** управляют ресурсами OpenGL (текстуры, буферы). Их оборачиваем в собственный RAII-класс: конструктор делает `glGen*`, деструктор — `glDelete*`. Тогда GPU-ресурс тоже освобождается сам. Это правило прямо закрывает утечки видеопамяти.

## 3. Обработка ошибок — «сломанное не ломает остальное»

Ключевая идея: функция должна **честно сообщать, что не смогла**, а вызывающий — решать, что делать. Плохо — когда функция при ошибке молча возвращает мусор или роняет всю программу.

Разделяем два вида ошибок.

**Ожидаемый сбой** (битый пакет пришёл, нет GPS-фикса, сервер недоступен) — это не исключительная ситуация, это нормальная жизнь. Возвращаем `std::optional<T>` (есть значение или нет) или `std::expected<T, Error>` (значение или причина ошибки, C++23). Вызывающий **обязан** проверить.

```cpp
// плохо — вернули -1 как «ошибку», легко забыть проверить, -1 уедет в расчёт
float parse_lap_time(const Packet& p);   // вернёт -1 при ошибке? кто это помнит?

// хорошо — тип заставляет проверить
std::optional<LapTime> parse_lap_time(const Packet& packet);

auto lap = parse_lap_time(packet);
if (!lap) {
    log_warn("Пакет без валидного времени круга, пропускаем");
    return;                  // сбой одного пакета не трогает остальные
}
use(*lap);
```

**Настоящая исключительная ситуация** (кончилась память, сломан инвариант, который никогда не должен ломаться) — исключение. Мы **разрешаем** исключения в client и track_server (десктоп), но с правилом: ловим их на границах модулей, не даём вырасти до краха всей программы. Обработка входящего сетевого пакета оборачивается в try/catch — один битый пакет логируется и отбрасывается, сервер живёт дальше.

Правило изоляции: **граница модуля — это стена.** Ошибка внутри парсера пакетов не должна течь наружу неконтролируемо. Она либо превращается в `optional`/`expected`, либо ловится на границе. Так один сломанный кусок остаётся сломанным локально.

Проверяем вход на входе. Публичная функция первым делом проверяет аргументы (длина буфера до чтения, не-null указатели). Дальше по коду уже доверяем.

## 4. Границы модулей и независимость

Это слой, что реально даёт «замену и независимость».

**Модуль показывает наружу минимум.** В заголовке (`.hpp`) — только то, чем реально пользуются снаружи. Всё остальное прячется в `.cpp`. Чем меньше показано, тем свободнее менять внутренности, не ломая других.

**Зависим от абстракций, а не от конкретики.** У вас это уже есть в дизайне: `DataSource` — абстракция, `ServerSource` и `LocalReceiverSource` — реализации. Остальной код работает с `DataSource` и не знает, какая под ним конкретно. Захотели третий источник — добавили класс, остального не тронули. Это и есть «лёгкая замена». Держите этот паттерн везде, где ждёте будущих вариантов.

**Однонаправленные зависимости.** `rajagp_core` не знает про `client` и `track_server` — он общий и ни от кого не зависит. `client` и `track_server` зависят от `core`, но не друг от друга. Стрелки зависимостей идут в одну сторону, циклов нет. Если потянуло сделать так, чтобы core что-то знал о клиенте — стоп, дизайн поехал.

**Одна функция — одна задача.** Если функция делает три вещи, её нельзя переиспользовать и тяжело тестировать. Признак беды: не можете назвать функцию одним глаголом, или в имени напрашивается «и» (`parse_and_send`). Разбейте.

## 5. Комментарии

Комментарий объясняет **почему**, а не **что**. Что делает код — видно из кода (если не видно — перепишите код, а не добавляйте комментарий).

```cpp
// плохо — пересказ кода, шум
i = i + 1;   // увеличиваем i на 1

// хорошо — объясняет неочевидное решение
// UM980 не держит SIGNALGROUP между перезагрузками, поэтому
// перечитываем конфигурацию при каждом старте (см. баг с COM2)
reload_signal_config();
```

Над каждой публичной функцией — короткий блок: что делает, что принимает, что возвращает, что при ошибке. Русский текст — можно и нужно, документ ваш.

```cpp
/// Разбирает 37-байтный TDMA-пакет.
/// Проверяет magic-marker и CRC-16 ДО чтения полей.
/// Возвращает пустой optional, если пакет битый (не бросает исключение).
std::optional<TelemetryFrame> parse_packet(std::span<const uint8_t> raw);
```

Помечаем то, к чему вернёмся, едиными метками, чтобы их можно было искать: `// TODO:` — доделать, `// FIXME:` — работает, но криво, `// HACK:` — временный костыль с объяснением почему. ИИ и человек ищут по этим словам.

## 6. Логирование

Используем spdlog, шесть уровней. Правило какой уровень когда:

- `trace` / `debug` — детали для отладки, в релизе выключены.
- `info` — нормальные события жизни («сервер запущен», «трекер N подключился»).
- `warn` — что-то не так, но работаем дальше (битый пакет отброшен).
- `error` — операция не удалась, но программа жива.
- `critical` — дальше работать нельзя.

Не логируем в горячем цикле на каждый кадр/пакет без нужды — утопите нужное в шуме и просадите скорость. Логируем события, а не поток.

## 7. Потоки и синхронизация

Сеть (WebSocket) и рендер (OpenGL) в разных потоках — самая опасная зона.

- **Общие данные — под защитой.** Любую переменную, которую пишет один поток и читает другой, либо защищаем `std::mutex`, либо передаём через потокобезопасную очередь. Никаких «просто общих» переменных между потоками.
- **Предпочитаем очередь мьютексу.** Сетевой поток кладёт готовый снимок данных в очередь, рендер забирает. Потоки не лезут в одну память одновременно — гонок нет по построению. Это чище, чем расставлять мьютексы.
- **OpenGL — только из своего потока.** Все `gl*`-вызовы в потоке, где создан контекст. Из сетевого потока GL не трогаем никогда.
- Держим захват мьютекса как можно короче — залочили, взяли данные, разлочили. Не держим лок во время тяжёлой работы.

## 8. Общий стиль на каждый день

- `const` по умолчанию. Переменная не меняется — помечаем `const`. Читатель сразу видит, что тут ничего не поменяется.
- `auto` — когда тип очевиден из правой части (`auto r = std::make_unique<Receiver>()`), но не когда он прячет важное (не `auto x = get()`, если непонятно, что вернулось).
- Магические числа — в именованные константы. `if (len < 37)` → `if (len < PACKET_SIZE_BYTES)`.
- Инициализируем переменные при объявлении. Неинициализированная переменная — источник трудноуловимых багов.
